## 1. 技术栈选型

### 1.1 OpenClaw 侧

| 层次 | 技术 | 理由 |
|------|------|------|
| Plugin 运行时 | TypeScript (Node.js) | OpenClaw Plugin 官方格式，类型安全 |

### 1.2 Unity 侧

| 层次 | 技术 | 理由 |
|------|------|------|
| HTTP Server | `System.Net.HttpListener` | .NET 内置，无第三方依赖 |
| WebSocket Server | Unity 2022.3+: `HttpListenerContext.AcceptWebSocketAsync`（内置）<br>Unity 2021.3: `websocket-sharp`（MIT，~200KB） | 编译/Console 事件实时推送，代码循环核心支撑 |

| 线程调度 | `EditorApplication.update` 队列 | Unity 官方推荐主线程调度方式 |
| 编译监听 | `CompilationPipeline.compilationFinished` | 捕获结构化编译错误 |
| Console 捕获 | `Application.logMessageReceived` | 运行时日志收集 |
| 脚本反射 | `AppDomain.CurrentDomain.GetAssemblies()` | 分析项目脚本 API，编译成功后可用 |
| 初始化 | `[InitializeOnLoad]` | Editor 启动时自动运行 |

## 2. 通信协议设计

### 2.1 协议选择

采用 **HTTP REST + WebSocket 双协议**，各司其职，分端口部署：HTTP 端口 23456，WebSocket 独立端口 23457（= HTTP 端口 + 1）。

| 协议 | 职责 | 理由 |
|------|------|------|
| **HTTP REST** | Agent 主动发起的操作（创建对象、写脚本、查询场景等） | 请求-响应天然匹配 Agent 工具调用模型，无状态，调试方便 |
| **WebSocket** | Unity 主动推送事件（编译完成/失败、Console 日志、PlayMode 变化） | 编译耗时不定（1~30s），轮询体验差；事件驱动是代码自修正循环的核心 |
| ~~命名管道~~ | - | 平台差异大，调试困难，不采用 |
| ~~文件监听~~ | - | 延迟高，竞争条件，不采用 |

**WebSocket 版本兼容策略**：

```
Unity 版本检测（运行时）
    ├── 2022.3+  →  HttpListenerContext.AcceptWebSocketAsync（.NET 内置）
    └── 2021.3   →  websocket-sharp（MIT License，~200KB，UPM 包引入）
```

条件编译隔离两套实现，对上层 Handler 透明，接口一致。

### 2.2 REST API 规范

**基础 URL**: `http://localhost:23456/api/v1`

**统一响应格式**:
```json
{
  "ok": true,
  "data": { ... },
  "error": null
}
```

**错误响应格式**:
```json
{
  "ok": false,
  "data": null,
  "error": {
    "code": "SCENE_NOT_LOADED",
    "message": "No scene is currently open in the editor",
    "details": {}
  }
}
```

**错误码体系**:

| 错误码 | HTTP 状态 | 说明 |
|--------|-----------|------|
| `SERVER_NOT_READY` | 503 | Unity Editor 未就绪 |
| `SCENE_NOT_LOADED` | 422 | 无打开的场景 |
| `OBJECT_NOT_FOUND` | 404 | 指定 GameObject 不存在 |
| `ASSET_NOT_FOUND` | 404 | 指定资产路径不存在 |
| `INVALID_PARAMS` | 400 | 参数格式错误 |
| `EXECUTION_FAILED` | 500 | Unity API 执行异常 |
| `COMPILE_ERROR` | 422 | 脚本编译失败 |
| `FILE_NOT_FOUND` | 404 | 指定路径的文件不存在 |
| `FILE_OUTSIDE_PROJECT` | 400 | 路径超出 Assets/ 范围，拒绝访问 |

### 2.4 WebSocket 事件规范

**连接地址**: `ws://localhost:23456/ws`

**事件消息格式**:
```json
{
  "event": "compile_complete",
  "timestamp": "2026-02-28T10:00:00Z",
  "data": { ... }
}
```

**事件类型列表**:

| 事件名 | 触发时机 | data 内容 |
|--------|---------|-----------|
| `compile_started` | 编译开始 | `{}` |
| `compile_complete` | 编译成功完成 | `{ duration_ms }` |
| `compile_failed` | 编译有错误 | `{ errors: [{ file, line, column, message }] }` |
| `console_log` | Console 新增日志 | `{ type: "log/warning/error", message, stackTrace }` |
| `playmode_changed` | Play Mode 状态变化 | `{ state: "entering/playing/exiting/stopped" }` |
| `scene_saved` | 场景保存完成 | `{ scenePath }` |
| `build_started` | 构建开始 | `{ platform }` |
| `build_complete` | 构建成功 | `{ platform, outputPath, duration_ms }` |
| `build_failed` | 构建失败 | `{ platform, errors: [...] }` |

## 3. Unity 侧关键技术问题

### 3.1 主线程调度

Unity 所有 Editor API 必须在主线程调用，而 HttpListener 在后台线程处理请求。

**解决方案**：

```csharp
// 主线程任务队列
private static readonly Queue<Func<object>> _mainThreadQueue = new();
private static readonly Queue<TaskCompletionSource<object>> _completionSources = new();

// EditorApplication.update 中消费队列
private static void ProcessMainThreadQueue() {
    while (_mainThreadQueue.Count > 0) {
        var task = _mainThreadQueue.Dequeue();
        var tcs = _completionSources.Dequeue();
        try {
            var result = task();
            tcs.SetResult(result);
        } catch (Exception ex) {
            tcs.SetException(ex);
        }
    }
}

// 后台线程提交任务（阻塞等待结果）
public static T RunOnMainThread<T>(Func<T> action) {
    var tcs = new TaskCompletionSource<object>();
    lock (_lock) {
        _mainThreadQueue.Enqueue(() => action());
        _completionSources.Enqueue(tcs);
    }
    return (T)tcs.Task.GetAwaiter().GetResult(); // 阻塞后台线程等待主线程执行
}
```

### 3.2 Editor 生命周期管理

Editor 的 Domain Reload 会销毁所有静态状态，需要在 `[InitializeOnLoad]` 中完整重建 Server：

```csharp
[InitializeOnLoad]
public static class UnityEditorServer {
    static UnityEditorServer() {
        // Domain Reload 后自动重新执行
        StopServer();   // 清理可能残留的旧监听器
        StartServer();
        EditorApplication.update += ProcessMainThreadQueue;
        AssemblyReloadEvents.beforeAssemblyReload += OnBeforeAssemblyReload;
    }

    static void OnBeforeAssemblyReload() {
        // 编译前关闭 Server，防止端口占用
        StopServer();
    }
}
```

### 3.3 端口冲突处理

若默认端口 23456 被占用，自动尝试 23457、23458...，并将实际端口写入文件供 Plugin 读取：

```
~/.openclaw/unity-editor-port   # 存储当前 Unity Server 端口
```

### 3.5 编译反馈捕获

`CompilationPipeline.compilationFinished` 回调提供完整的编译结果，在 Domain Reload 前捕获并存入静态缓存：

```csharp
// assemblyCompilationFinished：每个程序集编译完成时调用，携带 CompilerMessage[]
// compilationFinished：所有程序集编译完成后调用，用于发出汇总事件
[InitializeOnLoad]
public static class CompilationListener {
    private static readonly List<CompilerMessage> _messages = new();

    static CompilationListener() {
        // 注意：assemblyCompilationFinished 才有 CompilerMessage[]，compilationFinished 没有
        CompilationPipeline.assemblyCompilationFinished += OnAssemblyCompiled;
        CompilationPipeline.compilationStarted        += _ => _messages.Clear();
        CompilationPipeline.compilationFinished       += OnAllCompilationFinished;
    }

    static void OnAssemblyCompiled(string assemblyPath, CompilerMessage[] messages) {
        // 收集所有程序集的错误和警告
        _messages.AddRange(messages);
    }

    static void OnAllCompilationFinished(object context) {
        var errors = _messages.Where(m => m.type == CompilerMessageType.Error).ToArray();
        CompileResultCache.Store(_messages.ToArray());
        if (errors.Length > 0)
            EventBroadcaster.Broadcast("compile_failed",   new { errors });
        else
            EventBroadcaster.Broadcast("compile_complete", new { duration_ms = 0 });
    }
}
```

**降级行为**：编译失败时 `GET /project/scripts` 只返回文件路径列表，不做反射分析（因为 Assembly 未重载）。

### 3.6 Console 日志捕获

```csharp
static ConsoleLogger() {
    Application.logMessageReceivedThreaded += OnLogMessage; // 线程安全版本
}

static void OnLogMessage(string message, string stackTrace, LogType type) {
    _ringBuffer.Enqueue(new LogEntry { message, stackTrace, type, time = DateTime.UtcNow });
    EventBroadcaster.Broadcast("console_log", new { type, message, stackTrace });
}
```

使用定容环形缓冲区（默认 500 条），防止内存无限增长。

### 3.7 项目脚本反射分析

编译成功后通过反射枚举所有用户定义类型：

```csharp
// 只分析 Assets/ 下的程序集，排除 Unity 引擎程序集
var userAssemblies = AppDomain.CurrentDomain.GetAssemblies()
    .Where(a => a.GetName().Name.StartsWith("Assembly-CSharp"));

var result = userAssemblies
    .SelectMany(a => a.GetTypes())
    .Where(t => t.IsPublic)
    .Select(t => new {
        fullName  = t.FullName,
        baseType  = t.BaseType?.Name,
        methods   = t.GetMethods(BindingFlags.Public | BindingFlags.Instance | BindingFlags.DeclaredOnly)
                     .Select(m => m.ToString()),
        fields    = t.GetFields(BindingFlags.Public | BindingFlags.Instance)
                     .Select(f => new { f.Name, type = f.FieldType.Name }),
        isMonoBehaviour = typeof(MonoBehaviour).IsAssignableFrom(t)
    });
```

**性能说明**：首次调用约 50~200ms（取决于项目规模），结果缓存直到下次编译完成。

### 3.8 资源检索策略（关键字 + 元数据，不使用 RAG）

非文本资产（音频、贴图、动画等）通过以下方式检索，不引入向量数据库：

| 检索维度 | 实现方式 | 示例 |
|---------|---------|------|
| 类型过滤 | `AssetDatabase.FindAssets("t:AudioClip")` | 找所有音频文件 |
| 名称/路径 | `AssetDatabase.FindAssets("BGM t:AudioClip")` | 找名含 BGM 的音频 |
| 元数据过滤 | 加载 AssetImporter 读取属性 | 按时长、采样率筛选音频 |
| 依赖反查 | `AssetDatabase.GetDependencies()` + 反向索引 | 找哪些 Prefab 引用了某个音频 |
| 目录约定 | 直接按路径前缀搜索 | `Assets/Audio/BGM/` |

**约定优于配置**：推荐项目遵循语义化目录结构（`Audio/BGM/`、`Audio/SFX/`、`UI/Prefabs/` 等），配合关键字搜索可覆盖绝大多数资源定位需求。

### 3.4 GameObject 标识符问题

Unity 中 GameObject 无稳定 ID（`GetInstanceID()` 重启后变化），外部引用使用场景路径：

```
"Player/Body/Weapon"   → 层级路径，在同一场景内唯一
```

## 5. 安全性分析

| 威胁 | 风险等级 | 缓解措施 |
|------|----------|----------|
| 外网访问 Unity Server | 高 | HttpListener 仅绑定 `127.0.0.1` |
| 任意代码执行 | 高 | 不暴露通用 eval 接口，仅白名单操作 |
| 路径遍历 | 中 | 所有资产路径限制在 `Assets/` 目录下 |
| DoS (大量请求) | 低 | Server 单线程处理队列，天然限速 |
| Plugin 配置泄露 | 低 | Port 配置不包含敏感信息 |
